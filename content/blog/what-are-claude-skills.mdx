You use Claude every day. You paste context, explain your process, set up the same constraints. Every single time.

Skills change that.

## What is a skill?

A skill is a markdown file. Literally a `.md` file. It gives Claude specific instructions for performing a task. Your methodology, your process, your quality standards. Written once, used every time.

Think of it like a detailed brief that Claude follows automatically. Instead of re-explaining your process every session, you hand Claude the skill file and it picks up where you left off.

## What does a skill look like?

A skill has a few parts:

- **A name and description** so you (and Claude) know what it does
- **A trigger** that describes when to use it
- **Step-by-step instructions** for how to do the work
- **Quality criteria** so the output meets your standards

That's it. No code. No API keys. No complex setup. Plain text with some formatting.

## Why does this matter?

If you're using Claude for the same types of work repeatedly, you're wasting time re-prompting. Skills let you:

- **Encode your methodology** so Claude follows it consistently
- **Skip the setup** every session
- **Get better output** because the instructions are refined over time
- **Share your process** with team members who can use the same skill

The difference between prompting and skills is the difference between explaining your process every time and having it just run.

## Where to start

The fastest way to understand skills is to build one. Head to the [skill builder](/build) and answer a few questions about your workflow. You'll get a custom Skill.md in about 60 seconds.

Or browse the [resources page](/resources) to download skills, plugins, and connectors that are ready to use.
